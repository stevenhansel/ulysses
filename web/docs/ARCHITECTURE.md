# Ulysses Web — Architecture

> Version: 0.0.0 | Last updated: 2026-07-01

The Ulysses Web UI is a **React single-page application** that serves as the user interface for the Ulysses model manager proxy. It communicates with the Rust-based API backend (`api/`) over HTTP via a Vite dev proxy in development and direct API calls in production.

## Layered Architecture

The application follows a **layered architecture** with clear separation of concerns. Each layer has strict inward-only dependencies.

```
┌──────────────────────────────────────────────┐
│           Presentation Layer (Routes)          │
│  routes/  •  __root.tsx  •  index.tsx  •  ... │
│  Orchestrates features, provides layout        │
├──────────────────────────────────────────────┤
│              Feature Layer                     │
│  features/{domain}/                            │
│  ├── components/    UI components + tests      │
│  ├── api/           TanStack Query hooks       │
│  ├── forms/         Zod schemas                │
│  └── index.ts       Public barrel               │
├──────────────────────────────────────────────┤
│           Infrastructure Layer                 │
│  lib/  •  api-client.ts  (HTTP to Rust API)    │
│  hooks/  •  use-theme.ts  (theme management)   │
├──────────────────────────────────────────────┤
│              Shared Layer                      │
│  components/ui/   shadcn primitives            │
│  types.ts          Shared TS interfaces        │
│  lib/utils.ts     cn() helper                  │
└──────────────────────────────────────────────┘
```

### Presentation Layer (Routes)

Entry: `src/main.tsx`

The presentation layer is built on **TanStack Router with file-based routing**. Every `.tsx` file in `src/routes/` (except those prefixed with `-`) automatically becomes a route.

```typescript
// src/main.tsx — Application entry point
const queryClient = new QueryClient({ /* default options */ })
const router = createRouter({ routeTree, context: { queryClient } })

// Render with providers
<QueryClientProvider client={queryClient}>
  <RouterProvider router={router} />
  <ReactQueryDevtools />
</QueryClientProvider>
```

**Route table:**

| File | Path | Component | Description |
|---|---|---|---|
| `__root.tsx` | — | `RootLayout` | Navbar, `<Outlet />`, 404 handler |
| `index.tsx` | `/` | `HomePage` | Marketing landing, feature highlights |
| `dashboard.tsx` | `/dashboard` | `DashboardPage` | Hardware/inference metrics (stub) |
| `settings.tsx` | `/settings` | `SettingsPage` | Profile management (stub) |

The root layout (`__root.tsx`) provides the persistent shell:

```
<AppShell>
  <header>  ← Logo, nav links, theme toggle
    <nav>
      <Link to="/" />
      <Link to="/dashboard" />
      <Link to="/settings" />
      <ThemeToggle />
    </nav>
  </header>
  <main>    ← Route content renders here via <Outlet />
  </main>
</AppShell>
```

### Feature Layer

Each feature domain is a self-contained module under `src/features/{domain}/`. Features are **co-located by domain**, not by technical concern.

```
features/auth/
├── components/      ← UI components
│   ├── login-form.tsx
│   └── login-form.test.tsx
├── api/             ← Server state hooks (TanStack Query)
│   └── use-login.ts
└── index.ts         ← Public barrel (what other layers may import)

features/profiles/
├── components/
│   ├── profile-card.tsx
│   ├── profile-card.test.tsx
│   ├── avatar-uploader.tsx
│   └── avatar-uploader.test.tsx
├── api/
│   ├── use-profile.ts
│   └── use-profile.test.tsx
├── forms/           ← Zod schemas + React Hook Form types
│   └── profile-schema.ts
└── index.ts
```

**Feature module conventions:**

1. **`components/`** — Presentational and container components. Tests co-located.
2. **`api/`** — TanStack Query hooks (`useQuery`, `useMutation`) that call `apiClient`. Tests cover hook logic.
3. **`forms/`** — Zod validation schemas and inferred TypeScript types.
4. **`index.ts`** — Barrel file. Only what's exported here is part of the feature's public API. Internal implementation details stay private to the module.

**Rules:**
- Features may **not** import directly from other features' internal files — only via their `index.ts` barrel.
- Features **may** import from `components/ui/`, `lib/`, `hooks/`, and `types.ts`.

### Infrastructure Layer

#### API Client (`src/lib/api-client.ts`)

A thin fetch wrapper around the Rust API backend. All requests go through `/api/*` (proxied to `http://localhost:8000` in dev).

```typescript
// API conventions:
apiClient.get<T>('/profiles')
apiClient.get<T>('/profiles', { status: 'active' }) // query params
apiClient.post<T>('/auth/login', { username, password })
apiClient.put<T>('/profiles/1', { name: 'new-name' })
apiClient.delete('/profiles/1')
apiClient.upload<T>('/user/avatar', formData)
```

Error responses from the API are wrapped in `ApiClientError` which preserves the HTTP status and error message.

#### Theme Hook (`src/hooks/use-theme.ts`)

Manages light/dark mode using `useSyncExternalStore` for tear-free reads. Persists to `localStorage` and respects the system `prefers-color-scheme` as fallback. The `.dark` class is toggled on `<html>`.

```typescript
const { theme, setTheme, toggleTheme } = useTheme()
// theme: 'light' | 'dark'
```

### Shared Layer

#### UI Primitives (`src/components/ui/`)

Generated by **shadcn/ui** using the Radix-Nova style preset. These are unstyled, accessible primitives built on Radix UI. The current components are:

- `button.tsx` — Variants: default, secondary, outline, ghost, destructive, link
- `card.tsx` — Card, CardHeader, CardTitle, CardDescription, CardContent, CardFooter
- `dialog.tsx` — Dialog, DialogTrigger, DialogContent, DialogHeader, etc.
- `input.tsx` — Form input with error styling
- `label.tsx` — Form label

#### TypeScript Definitions (`src/types.ts`)

Placeholder for auto-generated TypeScript types from the Rust API using `ts-rs` (`#[derive(TS)]`). Currently contains manual definitions for:

- `LoginRequest`, `LoginResponse`, `User`
- `Profile`, `ProfileCreateRequest`, `ProfileUpdateRequest`
- `HardwareMetrics`, `CpuMetrics`, `GpuMetrics`, `RamMetrics`
- `InferenceMetrics`

## Routing & Navigation

### Route Generation

TanStack Router's Vite plugin scans `src/routes/` at build time and generates `src/routeTree.gen.ts`. This file:

- Imports each route module
- Calls `.update()` with the route's path and parent
- Constructs the `routeTree` by calling `_addFileChildren()`
- Generates full type information (`FileRoutesByFullPath`, `FileRouteTypes`, etc.)

### File Conventions

| Prefix/Suffix | Behavior |
|---|---|
| `__root.tsx` | Root route (layout, not a path) |
| `index.tsx` | Route at `/` |
| `dashboard.tsx` | Route at `/dashboard` |
| `-filename` | **Ignored** by the router generator (prefix with `-`) |

### Navigation

Use `<Link>` from TanStack Router for internal navigation:

```tsx
<Link to="/dashboard" className="text-muted-foreground">
  Dashboard
</Link>
```

Links are type-safe — the `to` prop only accepts valid route paths from the generated route tree.

## State Management

### Server State (TanStack Query)

All API data flows through TanStack Query. This provides:

- **Automatic caching** — Configurable `staleTime` (30s default)
- **Request deduplication** — Multiple components asking for the same data get one request
- **Background refetching** — Opt-in via `refetchOnWindowFocus` (disabled by default)
- **Optimistic updates** — Available via `onMutate` in mutations
- **Cache invalidation** — `queryClient.invalidateQueries()` on mutation success

```typescript
// Query pattern
function useProfiles() {
  return useQuery({
    queryKey: ['profiles'],
    queryFn: () => apiClient.get<Profile[]>('/profiles'),
  })
}

// Mutation with cache invalidation
function useCreateProfile() {
  const queryClient = useQueryClient()
  return useMutation({
    mutationFn: (data) => apiClient.post<Profile>('/profiles', data),
    onSuccess: () => queryClient.invalidateQueries({ queryKey: ['profiles'] }),
  })
}
```

### Client State

- **Theme** (`use-theme.ts`) — `useSyncExternalStore` with localStorage persistence
- **Form state** (react-hook-form) — Uncontrolled components, managed per-form

## Data Flow

```
User Interaction
      │
      ▼
Route Component (e.g., pages/dashboard.tsx)
      │
      ▼
Feature Component (e.g., features/profiles/ProfileCard)
      │
      ├── Form: react-hook-form + Zod validation
      │     │
      │     ▼
      │   apiClient.post('/profiles', validatedData)
      │
      └── Data: TanStack Query hook (useProfiles)
            │
            ▼
          apiClient.get('/profiles')
            │
            ▼
          Vite Dev Proxy (dev) / Direct (prod)
            │
            ▼
          Rust API (api/ — http://localhost:8000)
```

**Form submission flow (example — login):**

```
1. User types credentials
2. Zod schema validates on submit
3. react-hook-form manages field state & errors
4. useLogin mutation calls apiClient.post('/auth/login', data)
5. On success → onSuccess callback (e.g., redirect)
6. On error → ApiClientError caught, setError('root', ...)
7. Error message displayed in form
```

## Testing Strategy

### Levels

| Level | Tool | Scope | Location |
|---|---|---|---|
| **Component/UI** | Vitest + Testing Library | Render + user interactions | `*.test.tsx` next to component |
| **Hook/Logic** | Vitest + renderHook | Query/mutation behavior | `*.test.tsx` next to hook |
| **Route** | Vitest + RouterProvider | Full page rendering | `routes/*.test.tsx` |

### Mocking with MSW

All tests use **Mock Service Worker (MSW) v2** to intercept network requests at the HTTP level. This means:

- **No mocking of fetch/axios** — MSW intercepts real `fetch()` calls
- **Default handlers** in `tests/handlers.ts` provide sensible defaults (e.g., valid login with admin/password returns a token)
- **Per-test overrides** via `server.use()` allow customizing responses for edge cases

```typescript
// Override a handler for one test
server.use(
  http.post('/api/auth/login', () => {
    return HttpResponse.json(
      { message: 'Invalid credentials' },
      { status: 401 },
    )
  }),
)
```

### Test Infrastructure (`tests/`)

| File | Responsibility |
|---|---|
| `setup.ts` | Registers jest-dom matchers, starts/stops MSW server, mocks `window.matchMedia` |
| `mock-server.ts` | Creates `setupServer(...handlers)` from MSW/node |
| `handlers.ts` | Default mock handlers for all API endpoints |

### Running Tests

```bash
pnpm test          # Single run
pnpm test:watch    # Watch mode
```

## Key Design Decisions

### 1. File-based routing over manual routing

**Chosen:** TanStack Router file-based routing (Vite plugin auto-generates `routeTree.gen.ts`).

**Rationale:** Routes mirror the filesystem — no need to manually wire up route configuration. The tree regenerates on save during development.

### 2. Feature co-location over technical co-location

**Chosen:** Group files by domain feature (`features/auth/`, `features/profiles/`) not by technical role (`components/`, `hooks/`, etc.).

**Rationale:** When working on a feature, all related files are adjacent. Barrel files (`index.ts`) enforce a clean public API per feature.

### 3. fetch over axios for API client

**Chosen:** Native `fetch()` wrapped in a thin client.

**Rationale:** No additional dependency. Fetch is built into modern browsers and Node. The wrapper adds typed error handling and query parameter support.

### 4. Zod v3 compatibility via `zod/v3`

**Chosen:** Import from `zod/v3` to maintain compatibility with the `@hookform/resolvers` zod resolver.

**Rationale:** Zod v4 introduced a new internal API that broke some TypeScript inferences. The `zod/v3` subpath export provides the stable v3 API within the v4 package.

### 5. shadcn/ui with Radix-Nova style

**Chosen:** Radix-Nova preset with CSS variables for theming.

**Rationale:** Provides accessible, customizable primitives. CSS variables enable easy light/dark mode switching. Components are copied into the project (not a runtime dependency), allowing full customization.

### 6. MSW over mocking fetch/axios directly

**Chosen:** Mock Service Worker intercepts at the network level.

**Rationale:** Tests exercise the real `apiClient` code path. No need to mock specific HTTP libraries. Handlers are reusable across test files.

## Development Workflow

```bash
# From monorepo root:
pnpm web:dev       # Start Vite dev server (HMR at localhost:5173)
pnpm web:test      # Run all tests once
pnpm web:build     # Type-check + production build
pnpm web:lint      # Run oxlint

# From web/ directory:
pnpm dev           # Vite dev server
pnpm test          # Vitest single run
pnpm test:watch    # Vitest watch mode
pnpm build         # tsc -b && vite build
pnpm lint          # oxlint
pnpm preview       # Preview production build
```

### Adding a new route

1. Create `src/routes/my-page.tsx`
2. Export a route with `createFileRoute('/my-page')`
3. The Vite plugin auto-generates `routeTree.gen.ts`

### Adding a new feature

1. Create `src/features/{domain}/`
2. Add subdirectories: `components/`, `api/`, `forms/` (as needed)
3. Create `index.ts` barrel file
4. Import into a route

### Adding a new shadcn/ui component

```bash
npx shadcn@latest add dialog input label form
```

### Code generation (types from Rust)

When the Rust API uses `#[derive(TS)]`, regenerate TypeScript types:

```bash
cd api && cargo test export_types
# Types are written to web/src/types.ts
```
